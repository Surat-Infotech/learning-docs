# Caching, CDN, and Multi-Region Delivery

Latency is a product feature. The difference between a 40ms and a 400ms response is the difference between an app that feels native and one that feels like a website. At scale, that difference is decided less by your code and more by *where* responses are cached and *where* compute runs relative to data. This lesson covers the layers of caching that sit between Next.js and the user, and how to reason about multi-region delivery without introducing cross-region latency that erases all your gains.

## What You'll Learn

- How a CDN sits in front of Next.js and what it can and cannot cache
- The distinction between HTTP `Cache-Control` headers and Next's internal Data/Full Route caches
- Serving cached HTML at the edge with stale-while-revalidate
- Multi-region deployment models and the compute-near-data problem
- Cache-key segmentation pitfalls when personalization enters the picture
- Strategies for purging and invalidating content across a CDN
- When edge rendering helps and when it actively hurts

## The layers of caching

There is no single cache. A request passes through several, and confusion about which one you're tuning is the most common source of "why is it still stale" bugs.

```text
Browser cache
   │
CDN / edge cache        ← Cache-Control headers, SWR, purge API
   │
Next.js Full Route Cache (HTML/RSC payload, build + ISR)
   │
Next.js Data Cache      (fetch() results, revalidate, tags)
   │
Your origin (DB, upstream APIs)
```

The Next.js caches live inside your deployment. The CDN cache lives in front of it, in dozens of points of presence (PoPs) around the world. A cache miss at the CDN falls through to Next; a cache hit never touches your compute at all. That last point is the whole game: **the cheapest, fastest response is one that never reaches your server.**

## CDN in front of Next

When you deploy to Vercel, Cloudflare, or put a CDN like Fastly/CloudFront in front of a self-hosted Next app, static assets and statically-rendered pages are automatically distributed to the edge. Dynamic responses are not cached unless you tell the CDN they're cacheable.

The control surface is `Cache-Control`. You set it on Route Handlers and on responses; the CDN reads it.

```ts
// app/api/products/route.ts
import { NextResponse } from "next/server";

export async function GET() {
  const data = await getProducts();
  return NextResponse.json(data, {
    headers: {
      // Browser caches 60s; shared CDN cache 300s; serve stale for 1h while
      // revalidating in the background.
      "Cache-Control": "public, max-age=60, s-maxage=300, stale-while-revalidate=3600",
    },
  });
}
```

Key directives senior engineers should have memorized:

- `max-age` — how long the *browser* may use the response without revalidating.
- `s-maxage` — how long a *shared* cache (the CDN) may use it. Overrides `max-age` for CDNs.
- `stale-while-revalidate` (SWR) — after `s-maxage` expires, keep serving the stale copy to the user *immediately* while fetching a fresh one in the background. The next request gets fresh data. This is how you get cache-hit latency with near-fresh content.
- `private` — never store in a shared cache. Use for anything user-specific.

### Cache-Control vs the Next caches

These solve different problems and can conflict.

- **Next Data Cache** (`fetch(url, { next: { revalidate: 60 } })`) caches the *upstream fetch result* inside your deployment, deduplicated across requests. It does not put anything at the edge.
- **Next Full Route Cache** caches the rendered HTML/RSC for statically-rendered routes, regenerated by ISR.
- **`Cache-Control` / `s-maxage`** controls the *CDN* in front of everything.

For a page rendered with ISR (`export const revalidate = 60`), Next emits an appropriate `s-maxage`/SWR header automatically so the CDN and the route cache stay aligned. The trouble starts when you hand-set `Cache-Control` on a Route Handler *and* expect the Data Cache to also matter, or when you set a long `s-maxage` on a page that reads cookies and is therefore dynamic. Decide which layer owns freshness for a given route and don't fight yourself across two.

## Serving cached HTML at the edge

For content that's the same for everyone (marketing pages, docs, product listings), the ideal is: render once, cache the HTML at every PoP, serve from the nearest one. ISR gives you this with `stale-while-revalidate` semantics baked in.

```tsx
// app/blog/[slug]/page.tsx
export const revalidate = 3600; // regenerate at most hourly

export async function generateStaticParams() {
  const posts = await getPostSlugs();
  return posts.map((slug) => ({ slug }));
}

export default async function Post({
  params,
}: {
  params: Promise<{ slug: string }>;
}) {
  const { slug } = await params;
  const post = await getPost(slug);
  return <article>{/* ... */}</article>;
}
```

Here a request anywhere in the world hits the local PoP, gets a static HTML hit (single-digit ms), and the origin only does work once an hour. This is the pattern you should reach for by default and deviate from only when you have a personalization reason.

## Multi-region and the compute-near-data problem

When you move *compute* to the edge (running rendering or Route Handlers in many regions), you must ask: where does the data live? An edge function in Sydney that queries a Postgres primary in Virginia makes several round trips across the Pacific. You've moved compute closer to the user but *farther from the data*, and a page that does three sequential queries now pays 600ms of latency it didn't before.

```text
BAD:  User(Sydney) → Edge fn(Sydney) → DB(Virginia) ×3 queries  = ~600ms
GOOD: User(Sydney) → Edge fn(Sydney) → Read replica(Sydney)     = ~30ms
GOOD: User(Sydney) → Region(Virginia) near DB → cached HTML edge = fast
```

There are two correct shapes, and choosing between them is a senior judgement call:

1. **Data near compute.** Run regional read replicas (or a globally-distributed database like a distributed Postgres/edge KV) so the data is local wherever compute runs. Best when reads dominate and you can tolerate replica lag.
2. **Compute near data.** Pin your dynamic compute to the region holding the database, and rely on the *CDN* to put cached responses near users. Best when you have a single primary, strong-consistency needs, or write-heavy paths.

The common mistake is enabling the Edge runtime everywhere because it sounds faster, without checking the data path. **Edge runtime is fast to start and globally distributed, but it amplifies any cross-region data call.** For data-heavy dynamic pages, a single well-placed region next to the database often beats edge.

### Choosing a runtime

```tsx
// Default Node.js runtime: full APIs, runs in your deployment region(s).
// Good for data-heavy server components near the database.

// Edge runtime: opt in explicitly.
export const runtime = "edge";
// Good for: light middleware-like logic, geolocation, A/B routing,
// auth-token checks — work that needs no heavy DB access.
```

## Cache segmentation and personalization

The instant a response varies per user, naive CDN caching becomes dangerous: you can serve User A's data to User B. The cache key must include everything the response depends on.

```ts
// app/api/feed/route.ts
import { NextResponse } from "next/server";

export async function GET(req: Request) {
  // If the response depends on the auth cookie, you MUST NOT share-cache it.
  const res = NextResponse.json(await getPublicFeed());
  res.headers.set("Cache-Control", "public, s-maxage=120, stale-while-revalidate=600");
  // Tell the CDN this varies by language so it keys correctly.
  res.headers.set("Vary", "Accept-Language");
  return res;
}
```

Rules to live by:

- **Personalized responses must be `private` (or uncached).** Never put a per-user response behind a shared cache without a per-user cache key.
- **Use `Vary` sparingly and precisely.** `Vary: Cookie` effectively makes the response uncacheable (every cookie value is a new key). Prefer splitting into a cached public shell + a small dynamic personalized fragment.
- **Split the page.** The senior pattern is a fully-cached static shell with personalization injected by a small client component or a streamed dynamic hole, so 95% of the bytes still come from the edge.

```tsx
// app/page.tsx — cached shell + streamed personal hole
import { Suspense } from "react";

export default function Page() {
  return (
    <>
      <StaticHero />            {/* cached at the edge for everyone */}
      <Suspense fallback={<GreetingSkeleton />}>
        <PersonalGreeting />     {/* dynamic, per-user, streamed */}
      </Suspense>
    </>
  );
}
```

## Purging and invalidation across a CDN

Invalidation is famously one of the two hard problems. Across a distributed CDN it's harder because you're invalidating in many PoPs at once. You have three tools, in increasing precision:

1. **Time-based (TTL/SWR).** Simplest, eventually consistent. Good default; content is stale for at most `s-maxage`.
2. **Tag-based revalidation.** Tag fetches and revalidate by tag when the underlying data changes. This is the Next-native, surgical approach.
3. **Path/URL purge.** Hit the CDN's purge API for specific URLs after a publish event.

```ts
// app/products/[id]/page.tsx
async function getProduct(id: string) {
  const res = await fetch(`${API}/products/${id}`, {
    next: { tags: [`product:${id}`] }, // tag this data
  });
  return res.json();
}

// app/actions.ts — invalidate precisely on write
"use server";
import { revalidateTag, revalidatePath } from "next/cache";

export async function updateProduct(id: string, data: FormData) {
  await saveProduct(id, data);
  revalidateTag(`product:${id}`);  // purge only what changed
  // revalidatePath("/products");  // or purge a whole route
}
```

A practical caution: `revalidateTag`/`revalidatePath` clear Next's own caches. Whether they also purge an *external* CDN in front of Next depends on the platform. On Vercel they're integrated; with a self-hosted CDN you must also call the CDN's purge API on publish, or accept TTL-based staleness. Map out your invalidation across *every* layer, not just the Next one.

### Trade-offs: when to use what

- **Static + ISR (cached HTML at edge)** — Default for shared content. Fast, cheap, resilient. Use unless personalization or real-time freshness forbids it.
- **Dynamic in a single region near data** — Use for data-heavy, write-heavy, or strongly-consistent pages. Lean on the CDN for the cacheable parts.
- **Edge runtime everywhere** — Use for light, data-light global logic (routing, geo, auth checks). Avoid for anything that makes multiple DB calls to a distant primary.
- **Tag-based invalidation** — Use when correctness matters and you can wire revalidation into your write paths. Falls back to TTL when you can't.
- **Long TTL with SWR** — Use to maximize hit rate when slightly-stale content is acceptable, which is more often than teams assume.

## Pitfalls

- **Caching personalized responses in a shared cache.** Leaks one user's data to another. Mark per-user responses `private` or give them a per-user key.
- **`Vary: Cookie`.** Makes responses effectively uncacheable. Split into cached shell + dynamic fragment instead.
- **Edge runtime with a distant database.** Multiplies cross-region latency. Check the data path before going edge.
- **Confusing the Data Cache with the CDN cache.** `revalidate` on a fetch does not put anything at the edge; `s-maxage` does. Know which layer owns freshness.
- **Forgetting external CDN purge.** Next's `revalidateTag` may not touch a self-hosted CDN. Wire purge into every layer.
- **Permanent caching of HTML.** Long `s-maxage` with no revalidation strands users on stale pages after a deploy. Always pair with SWR or invalidation.
- **Assuming edge is always faster.** Cold start and distance-to-data can make a single well-placed region win.

## Exercises

1. Add `Cache-Control` with `s-maxage` and `stale-while-revalidate` to a public Route Handler, then observe `age` and `x-cache` (or `x-vercel-cache`) headers across repeated requests to confirm CDN hits.
2. Build a page as a cached static shell with a `Suspense`-wrapped per-user fragment. Verify the shell is cached at the edge while the fragment renders dynamically.
3. Tag a data fetch and write a Server Action that calls `revalidateTag` on update. Confirm only the tagged route refreshes while sibling routes stay cached.
4. Diagram (and document) your app's data path for a hypothetical user in a far region. Identify whether you have a compute-near-data or data-near-compute problem and propose a fix.
5. Deliberately set `Vary: Cookie` on a cacheable endpoint, measure the collapse in hit rate, then refactor to a cached-shell pattern and compare.

## Key Takeaways

- The fastest response is one served from the edge that never reaches your compute; design for cache hits.
- `Cache-Control`/`s-maxage` controls the CDN; Next's Data and Full Route caches live inside your deployment. They are different layers.
- `stale-while-revalidate` gives you cache-hit latency with near-fresh content and should be a default.
- Multi-region only helps if data is local to compute; edge runtime amplifies cross-region data calls.
- Personalized responses must never share a cache key; split pages into cached shells plus dynamic holes.
- Invalidate across every layer — tag-based for precision, TTL/SWR as the resilient fallback, plus an explicit purge for external CDNs.
